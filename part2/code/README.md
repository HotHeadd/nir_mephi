# memwatch — опытный образец модуля перехвата событий доступа к памяти

Прототип внешнего агента, который ставит под охрану объявленную область памяти
чужого процесса и регистрирует недопустимые обращения к ней с атрибуцией к
инициатору (поток, модуль, функция). Реализован на связке `ptrace` + `mprotect`
в пользовательском пространстве, на немодифицированном ядре Linux (x86-64).

Это **минимально работоспособный** образец для проверки гипотез о
работоспособности и накладных расходах выбранного механизма, а не
полнофункциональный инструмент.

---

## Что делает

Агент запускает целевую программу под трассировкой, снимает право записи с
охраняемой области (через инжекцию `mprotect` в адресное пространство цели) и
ждёт нарушений. При попытке записи в область возникает аппаратное нарушение
прав; агент перехватывает его, определяет адрес инструкции-инициатора, по
картам памяти и таблицам символов ELF восстанавливает модуль и функцию, и
регистрирует событие. После этого охрана восстанавливается, и цель продолжает
исполнение.

Границы текущего прототипа:

- контролируется только запись (W); чтение и исполнение — направления развития;
- режим запуска цели под агентом (launch); подключение к работающему процессу
  (attach) не реализовано;
- модель цели однопоточная;
- реакция на нарушение — регистрация события, без пресечения доступа.

---

## Требования

- Linux, архитектура x86-64;
- `gcc` и `make`;
- право трассировать процессы (`ptrace`). Запуск из-под обычного пользователя
  для режима launch обычно достаточен; если система ограничивает трассировку,
  см. раздел «Отладка».

## Сборка

Собрать всё (агент, модельные цели, сценарии):

```
make
```

Очистить артефакты сборки:

```
make clean
```

Отдельные группы целей:

```
make              # всё сразу: агент, цели, сценарии
make target       # только модельные цели -> bin/target (PIE) и bin/target_nopie
make scenarios    # сценарии валидации -> bin/s1_... bin/s5_...
```

Отдельные группы `target` и `scenarios` не собирают агента — для самого агента
запускайте `make` (он входит в цель `all`).

Флаги сборки (в `Makefile`):

- агент — `-std=c11 -Wall -Wextra -g -O0 -D_GNU_SOURCE`;
- цели и сценарии — `-std=c11 -g -O0` (без `strip`, чтобы символы были доступны
  атрибуции); `bin/target_nopie` и `bin/s4_buffer_overflow` дополнительно
  собираются с `-no-pie` (фиксированный адрес, см. ниже).

Зависимости от заголовков отслеживаются автоматически (`-MMD -MP`), поэтому
правка `.h` пересобирает нужные объектные файлы без `make clean`.

## Запуск

Общая форма:

```
./bin/memwatch [параметры охраны] [--len N] [--log ФАЙЛ] [--allow СПИСОК] -- ПРОГРАММА [АРГ...]
```

Всё, что идёт после `--`, — это целевая программа и её аргументы.

### Флаги

| Флаг          | Назначение                                                            |
|---------------|-----------------------------------------------------------------------|
| `--symbol ИМЯ`| охраняемая область задаётся именем символа (приоритетный способ)      |
| `--addr 0xA`  | либо абсолютным виртуальным адресом                                    |
| `--len N`     | размер области; для `--symbol` необязателен (берётся из ELF), для `--addr` нужен |
| `--log ФАЙЛ`  | журнал событий в файл (по умолчанию — stderr)                          |
| `--allow Ф1,Ф2`| функции, запись из которых считается легитимной и не регистрируется  |
| `--`          | разделитель: далее идёт целевая программа и её аргументы               |

`--symbol` и `--addr` взаимоисключающи; нужно задать ровно один из них.

Размер области для `--symbol` по умолчанию берётся из таблицы символов ELF
(поле размера символа), поэтому `--len` указывать не обязательно. Явный `--len`
переопределяет это значение. Для `--addr` размер из адреса не выводится, поэтому
`--len` нужен (иначе используется значение по умолчанию 8 байт).

Замечание о `--addr`: абсолютный адрес пригоден только для целей, собранных с
`-no-pie` (фиксированная раскладка). Для обычных PIE-программ адрес меняется от
запуска к запуску из-за ASLR, поэтому используйте `--symbol`.

### Примеры

Охрана переменной по символу в базовой цели (адрес плавающий, PIE):

```
./bin/memwatch --symbol critical_flag -- ./bin/target
```

Запись журнала в файл:

```
./bin/memwatch --symbol critical_flag --log events.log -- ./bin/target
```

Признать запись из функции инициализации легитимной (не регистрировать её):

```
./bin/memwatch --symbol critical_flag --allow init_flag -- ./bin/target
```

Охрана по абсолютному адресу (для no-pie цели; адрес печатает сама программа):

```
./bin/s4_buffer_overflow                 # узнать адрес поля
./bin/memwatch --addr 0x404038 --len 8 -- ./bin/s4_buffer_overflow
```

## Сценарии валидации

Сценарии — это отдельные подопытные программы, запускаемые под агентом.

| Сценарий              | Что проверяет                          | Запуск                                                                           |
|-----------------------|----------------------------------------|----------------------------------------------------------------------------------|
| `s1_legit_write`      | легитимная запись                      | `./bin/memwatch --symbol critical_flag -- ./bin/s1_legit_write`          |
| `s2_foreign_module`   | атрибуция до чужого модуля (.so)       | `./bin/memwatch --symbol critical_flag -- ./bin/s2_foreign_module`       |
| `s3_foreign_func`     | атрибуция до чужой функции             | `./bin/memwatch --symbol critical_flag -- ./bin/s3_foreign_func`         |
| `s4_buffer_overflow`  | переполнение соседнего буфера          | `./bin/memwatch --addr <адрес> --len 8 -- ./bin/s4_buffer_overflow`              |
| `s5_control_unguarded`| избирательность (необращение)          | `./bin/memwatch --symbol critical_flag -- ./bin/s5_control_unguarded`    |

Ожидаемое поведение:

- `s1` без `--allow` регистрирует запись как событие, поскольку охрана
  устанавливается до инициализации; с `--allow main` событие не возникает;
- `s2` регистрирует событие с атрибуцией к библиотеке-нарушителю
  (`libs2attacker.so`, функция `attacker_write`);
- `s3` регистрирует событие с атрибуцией к функции `writer_foreign`;
- `s4` регистрирует событие: запись за границу буфера затирает соседнее поле;
- `s5` не регистрирует ни одного события: программа пишет в незащищённую
  переменную, охраняемая область не затрагивается.

## Формат вывода

Каждое зарегистрированное событие выводится в виде:

```
событие #1 | TID=714 | доступ=W
  адрес обращения : 0x55ecb099f018
  инструкция (RIP): 0x55ecb099c173
  модуль          : /путь/к/модулю + 0x1173
  функция         : illegitimate_write
```

Поля: порядковый номер события, идентификатор потока, тип доступа, адрес
обращения к охраняемой области, адрес инструкции-инициатора, модуль и смещение
внутри него, имя функции. Если у модуля нет таблицы символов (например, после
`strip`), строка функции принимает вид `<не разрешена (нет символов)>`, а
атрибуция ограничивается модулем и смещением.

## Принцип работы

Однократная подготовка:

1. запуск цели под трассировкой (`fork` + `PTRACE_TRACEME` + `execve`); первая
   остановка происходит на `execve`;
2. поиск в исполнимом отображении цели готовой инструкции `syscall` (опкод
   `0f 05`) — она нужна для инжекции;
3. разрешение охраняемой области: символ сопоставляется с фактическим адресом по
   картам памяти и таблицам символов ELF (либо берётся заданный `--addr`);
4. установка охраны — снятие права записи с области переводом её страниц в режим
   только для чтения.

Снятие прав выполняется инжекцией `mprotect`: внешний процесс не может изменить
права чужого адресного пространства напрямую, поэтому агент сохраняет регистры
цели, временно подменяет указатель инструкций на найденную `syscall` и
аргументы вызова, исполняет один шаг и восстанавливает регистры.

Цикл наблюдения:

1. цель исполняется до возникновения нарушения;
2. при записи в область только для чтения возникает `SIGSEGV`, и управление
   получает агент;
3. по адресу обращения событие классифицируется: попадание в сам объект —
   регистрируется; попадание в соседние данные той же страницы (следствие
   страничной гранулярности `mprotect`) — пропускается без регистрации; нарушение
   вне охраняемой страницы — пробрасывается цели;
4. для зарегистрированного события считывается контекст (адрес инструкции,
   идентификатор потока) и выполняется атрибуция к модулю и функции; если функция
   входит в белый список (`--allow`), обращение признаётся легитимным;
5. охрана восстанавливается: право записи временно возвращается, исполняется
   одна инструкция в пошаговом режиме (нарушившая запись завершается), затем
   право записи снова снимается;
6. цикл повторяется до завершения цели.

## Отладка

Если агент не перехватывает обращения или цель сразу падает, проверьте
ограничения трассировки в системе:

```
cat /proc/sys/kernel/yama/ptrace_scope
```

Значение `0` разрешает трассировку потомков без дополнительных привилегий.
Поскольку агент сам порождает цель (режим launch), этого обычно достаточно; при
более строгих политиках может потребоваться запуск с повышенными привилегиями.

Проверить, что нужный символ присутствует в цели:

```
nm ./bin/target | grep critical_flag
```

Способ задания области (`--symbol` или `--addr`) на атрибуцию не влияет: имя
функции-инициатора определяется по адресу инструкции при нарушении, независимо
от того, как объявлена охраняемая область.

И разрешение области по имени, и атрибуция функции опираются на таблицу
символов ELF. Поэтому у застрипанной цели (собранной со `strip`) отсутствует
и то, и другое по одной причине — символов в файле нет: охрану приходится
задавать через `--addr`, а атрибуция ограничивается модулем и смещением.
